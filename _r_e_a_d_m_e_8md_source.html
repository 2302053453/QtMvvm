15 # Features {#qtmvvm_readme_label_0}
16 The main feature of QtMvvm is the seperation between ui and logic. With this library, you can create a core library, containing your application logic, as well as ui controllers (called "ViewModels"), and create multiple ui projects on top of it. This way you can for example provide both, a widgets and a qt quick based application, or create different uis for different devices, without having to code anything twice.
17 
18 The key features are:
19 
20 - Create ViewModels in the core application to prepare data for presentation without binding to any concret GUI
21  - Supports singleton ViewModels
22  - Supports automatic presentation of container ViewModels
23 - Functions to show messageboxes (info, warning, error, etc.) from your core app
24  - Asynchronous, with result handling
25  - Supports input dialogs and native file dialogs out of the box
26  - Supports native file pickers on Android
27  - Supports color picker dialog
28  - Supports progress and busy indicator dialogs
29  - custom dialog types can be created
30 - Methods to create Two-Way Bindings from C++ and QML
31 - Macros and a ServiceRegistry to make Dependency Injection possible for Services and ViewModels
32 - A generic Presenter Interface so you can create your own custom GUI implementations
33  - Widgets and Qt Quick Controls 2 are supported out of the box
34 - Generic Edit-View factories to create simple edits for any kind of data from the core code
35 - Supports an extensive "Settings GUI" via a simple XML format

119 ## Adding new ViewModels and Views {#qtmvvm_readme_label_9}
120 The most important part is to know how to add new ViewModels and Views.
121 
122 ### Create the ViewModel {#qtmvvm_readme_label_10}
123 - Add a new c++ class to your core project. Let it inherit from `QtMvvm::ViewModel`
124 - Make shure the Constructor has the following signature: `Q_INVOKABLE MyClass(QObject *parent);`
125 - See [`examples/mvvmcore/SampleCore/sampleviewmodel.h`](examples/mvvmcore/SampleCore/sampleviewmodel.h) for an example ViewModel
126 
127 ### Create the View for QtWidgets {#qtmvvm_readme_label_11}
128 - Create a new widget class in your widgets gui project.
129 - In order to use it as widget for a viewmodel, you should name it accordingly: If your viewmodel is named `MyCustomViewModel`, the widgets name must start with `MyCustom` as well (e.g. `MyCustomWindow`, `MyCustomDialog`, `MyCustomView`, etc.)
130 - Modify the constructor to look like this: `Q_INVOKABLE MyCustomWindow(QtMvvm::ViewModel *viewModel, QWidget *parent = nullptr);`
131 - Create a member variable with your viewmodel type, e.g. `MyCustomViewModel *_viewModel;`
132 - In the constructors implementation, cast the `viewModel` to your viewmodel class and assign it to `_viewModel`
133 - As final step, you need to register the widget. This can be done by adding the line `WidgetPresenter::registerView<MyCustomWindow>();` to your `main.cpp`, before calling `QApplication::exec`
134 - See [`examples/mvvmwidgets/SampleWidgets/sampleview.h`](examples/mvvmwidgets/SampleWidgets/sampleview.h) for an example widget
135 
136 ### Create the quick ui {#qtmvvm_readme_label_12}
137 - *Optional:* Register the viewmodel for qml. This way autocomplete will work. Add the line `qmlRegisterUncreatableType<MyCustomViewModel>("com.example.mvvmexample", 1, 0, "MyCustomViewModel", "ViewModels cannot be created!");` to your main cpp before creating the engine.
138 - Create a new qml file inside of the folder `:/qml/views` (This is required to automatically find the view).
139 - In order to use it as view for a viewmodel, you should name it accordingly: If your viewmodel is named `MyCustomViewModel`, the views name must start with `MyCustom` as well (e.g. `MyCustomView`, `MyCustomActivity`, etc.)
140 - Add the following imports to the qml file:
141 ```qml
142 import QtQuick 2.10
143 import QtQuick.Controls 2.3
144 import de.skycoder42.QtMvvm.Core 1.1
145 import de.skycoder42.QtMvvm.Quick 1.1
146 import com.example.mvvmexample 1.0 //adjust to the module defined in your main.cpp
147 ```
148 - Add a property named viewmodel to the root element: `property MyCustomViewModel viewmodel: null` (If you did not register the viewmodel, use `var` instead of `MyCustomViewModel` as property type)
149 - Add a presenter progress element to the root element. This way lazy loading views will show a progress: `PresenterProgress {}`
150 - See [`examples/mvvmquick/SampleQuick/SampleView.qml`](examples/mvvmquick/SampleQuick/SampleView.qml) for an example view

152 # Understanding how QtMvvm works {#qtmvvm_readme_label_13}
153 The general idea is the following: You create viewmodels in your core project, which represent uis. They typically contain all the properties relevant for the ui, methods (slots) that can be called (e.g. on a button click) and signals to inform the ui about changes and other events. Thus, you can use the as ui "placeholders". Of course, They only contain the ui logic, not the actual uis.
154 
155 The CoreApp is what's reponsible for managing those viewmodels. Showing a viewmodel, as well as messages (alert dialogs) are all controlled by the coreapp. The coreapp uses a so called presenters to create the actual uis. The presenters are located in the ui projects and they are the most complicated part. Their main task is to find ui implementations for viewmodel (called views), and manage the life cycle as well as the presentation of those real views. The presenters are where the decision is made, how a specific view should be shown.
156 
157 The views are whatever you need to create actual uis. This depends on the presenter used, since the presenter selects the views. Each ui type has their own way to create those views, but they all have in commmon that the views themselves do not control the application. When a viewmodel gets shown, a new view is created and the viewmodel passed to it. Once the views was closed, the view and the viewmodel get deleted again. Views and ViewModels are temporary and should only use the viewmodel to interact with other parts of the application.

159 ## A side note on presenters {#qtmvvm_readme_label_14}
160 To create a presenter, the `QtMvvm::IPresenter` must be implemented and provided via the `ServiceRegistry`. Presenters can become quite complicated, but they are the thing you need to modify if you want to present views in a different way from the ones supported. Currently, the presenters can do the following:
161 
162 - Widgets Presenter
163  - Present viewmodels as widgets (or windows, dialogs, depending on type and parent)
164  - Parent-Aware: If your viewmodel has a parent and shows a ui, the childs ui will be a child of the parents ui as well
165  - Automatically detect QDockWidgets and place them as dock inside a parents QMainWindow
166  - Automatically detect QMdiSubWindows and place them in a parents QMdiArea
167  - Allows windows to implement `IPresentingWidget`. This way a window can handle the presentation of it's children without modifying the presenter
168  - Allows to register custom input widgets for input dialogs (and other parts, like the settings. See `InputWidgetFactory`)
169 - Quick Presenter
170  - Consists of a c++ part and a qml part
171  - The c++ part is responsible for finding views for controls and creating them, but **not** for the actual presentation. This is done by the qml presenter
172  - Allows to register custom input views for input dialogs (and other parts, like the settings. See `InputViewFactory`)
173  - The qml presenter can be any qml type. It must provide the correct methods (as seen in `QtMvvmApp`) and register itself with `Component.onCompleted: QuickPresenter.qmlPresenter = self`
174  - The `QtMvvmApp` qml types automatically register themselves as presenter and perform the presentations
175  - Supports Items as new fullscreen pages inside a stack view, as drawer or as tabs
176  - Supports Popups as modal dialogs
